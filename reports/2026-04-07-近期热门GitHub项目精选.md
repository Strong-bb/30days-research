## 一、筛选标准（怎么判断「真有用」）

我用了四个硬指标来过滤项目：

| 指标 | 含义 | 权重 |
|------|------|------|
| **GitHub 数据** | Star 数、Fork 数、Issue 活跃度、提交频率 | 高 |
| **HN 社区讨论** | 是否有真实用户在 Hacker News 上讨论？评论质量如何？ | 高 |
| **解决真问题** | 是否解决开发者日常遇到的实际痛点？还是为了 AI 而 AI？ | 高 |
| **代码质量** | 项目结构是否合理？文档是否完整？是否只是周末 hack？ | 中 |

**特别说明**：2026 年 3 月出现了大量「Claude Code skill」类项目。这些本质上是 Markdown 文件（给 AI agent 的指令集），质量参差不齐。我只收录了经过社区验证、确实有实际价值的。

## 二、AI/LLM 工具类

### 1. Karpathy/autoresearch — 让 AI 自主做研究

- **GitHub**: https://github.com/karpathy/autoresearch
- **Star 数**: 67,334（一个月内）| Fork: 9,689 | Language: Python | License: MIT
- **创建时间**: 2026-03-06

**一句话**：给 AI agent 一个小型 LLM 训练环境，让它自己修改代码、训练 5 分钟、检查结果、保留或丢弃，循环往复——你早上醒来就有一堆实验日志。

**为什么不是噱头**：
- Karpathy 本人的项目，他的 nanogpt、llm.c 都是社区标杆
- HN 上 237 分的扩展讨论帖，有人真的在 GPU 集群上跑了（SkyPilot 博客详述了规模化经验）
- 设计极度克制：只改一个文件 `train.py`，固定 5 分钟时间预算，指标明确（val_bpb）
- 不是通用 AI agent 框架——它只做一件事，但做得非常纯粹

**怎么装**：
```bash
# 需要：NVIDIA GPU（H100 测试过）、Python 3.10+、uv
curl -LsSf https://astral.sh/uv/install.sh | sh
git clone https://github.com/karpathy/autoresearch.git
cd autoresearch
uv sync
uv run prepare.py   # 一次性数据准备
uv run train.py      # 手动跑一次看看
# 然后让 Claude/Codex 在这个 repo 里自主跑
```

**适用场景**：
- 想了解 AI 如何自主做实验的研究者
- 单 GPU 就能跑，适合个人学习
- LLM 训练教学的绝佳入门材料

**坑在哪**：
- 必须 NVIDIA GPU，Mac/Linux 需要用社区 fork 版本（README 里列了 macOS、Windows、AMD 版）
- Karpathy 自己说「不确定是否要支持 CPU/MPS」，不会膨胀代码
- 需要 Claude 或 Codex 的 API 费用（agent 自主修改代码那部分）

/plugin install codex@openai-codex
/reload-plugins
/codex:setup

# 需要预先安装 Codex：

**适用场景**：

- **Star 数**: 3,206 | Fork: 115 | Language: Swift | License: MIT
- **HN 讨论热度**: 732 分、153 条评论

**一句话**：把 macOS 26 Tahoe 里内置的 Apple Intelligence 本地 LLM 暴露为 CLI 和 OpenAI 兼容的 HTTP 服务器——零费用、零 API key、完全离线。

**为什么不是噱头**：
- HN 732 分，是近一个月最火的 Show HN 之一
- 真正利用了硬件自带能力：Apple Silicon Mac 已经有了 LLM，但 Apple 只让你通过 Siri 用
- 支持 `echo "xxx" | apfel` 的 UNIX 管道模式，也支持 `apfel --serve` 变成 OpenAI 兼容服务器
- Tool calling 支持，可以和 agent 框架集成

**怎么装**：
```bash

brew tap Arthur-Ficial/tap
brew install apfel

# 或从源码编译：
git clone https://github.com/Arthur-Ficial/apfel.git
cd apfel && make install
```

**适用场景**：
- Mac 用户想要完全离线、零成本的本地 LLM

**坑在哪**：
- 只支持 macOS 26 Tahoe 及以上版本
- 上下文窗口只有 4096 token
- 必须是 Apple Silicon（Intel Mac 不行）
- 模型能力有限，不适合复杂推理任务
- 必须在系统设置里启用 Apple Intelligence

**适用场景**：

### 2. adammiribyan/zeroboot — 亚毫秒级 VM 沙箱

- **GitHub**: https://github.com/adammiribyan/zeroboot
- **Star 数**: ~311（HN 讨论热度） | Language: Rust | License: Apache-2.0
- **HN 讨论热度**: 311 分、72 条评论

**一句话**：用 Firecracker + Copy-on-Write 内存分叉，在 0.79ms 内创建一个真正的 KVM 虚拟机沙箱——比 E2B 快 190 倍，内存占用仅为 1/500。

**为什么不是噱头**：
- HN 311 分，技术社区对底层基础设施创新的认可
- 解决 AI agent 代码执行的安全隔离问题——这是当前 agent 框架的刚需
- 性能数据公开透明：spawn p50 0.79ms vs E2B ~150ms，内存 265KB vs 128MB
- 每个沙箱是真正的硬件级隔离（KVM VM），不是容器层面的隔离

**怎么装**：
```bash
# 需要支持 KVM 的 Linux 主机
git clone https://github.com/adammiribyan/zeroboot.git
# 参见 docs/DEPLOY.md

# 或试用公开 API：
curl -X POST https://api.zeroboot.dev/v1/exec \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer zb_demo_hn2026' \
  -d '{"code":"import numpy as np; print(np.random.rand(3))"}'
```

**适用场景**：
- AI agent 代码执行的安全沙箱
- 需要极快速、极轻量的隔离环境
- 替代 E2B、microsandbox 等商业方案

**坑在哪**：
- 还是「working prototype」，尚未生产级稳定
- 需要 KVM 支持的 Linux 主机（Mac/Windows 不能直接跑）
- 沙箱内没有网络（只能串口 I/O）
- CSPRNG 状态共享问题——numpy、OpenSSL 等需要在 fork 后手动重置随机种子
- 每个 fork 只有单 vCPU

### 3. onecli/onecli — AI Agent 的凭证保险箱

- **GitHub**: https://github.com/onecli/onecli
- **Star 数**: 1,671 | Fork: 80 | Language: TypeScript | License: Apache-2.0

**一句话**：开源的凭证网关——存一次 API key，AI agent 通过网关调用时自动注入，agent 永远看不到真实密钥。

**为什么不是噱头**：
- 解决了 AI agent 生态的真实安全问题：每个 agent 都需要各种 API key，明文存储风险极大
- 架构合理：Rust 网关 + Next.js 管理面板 + AES-256-GCM 加密存储
- Agent 只拿到假 key（如 `FAKE_KEY`），网关在运行时替换为真实凭证
- 支持 Bitwarden 集成，可以按需从密码管理器获取凭证

**怎么装**：
```bash
# 一键安装：
curl -fsSL https://onecli.sh/install | sh

# 或 Docker：
git clone https://github.com/onecli/onecli.git
cd onecli
docker compose -f docker/docker-compose.yml up -d --wait
# 打开 http://localhost:10254
```

**适用场景**：
- 管理多个 AI agent 的 API 凭证
- 团队协作中统一管控 agent 的权限
- 安全合规要求高的企业环境

**坑在哪**：
- MITM 方式拦截 HTTPS 请求——在某些企业网络环境下可能有问题
- 本地单用户模式没有登录，多人使用需要配置 Google OAuth
- 相对年轻的项目（不到一个月）

### 1. chenglou/pretext — 告别 DOM 测量的性能杀手

- **GitHub**: https://github.com/chenglou/pretext
- **Star 数**: 40,458 | Fork: 2,155 | Language: TypeScript | License: MIT
- **创建时间**: 2026-03-07

**一句话**：纯 JS/TS 实现的多行文本测量和布局库——不需要触发 DOM reflow 就能精确计算文本高度、行数、行宽。

**为什么不是噱头**：
- 作者 chenglou 是 ReasonML/Revery 社区知名开发者，技术实力过硬
- 解决了前端性能优化的经典难题：`getBoundingClientRect` 触发 reflow 是浏览器里最昂贵的操作之一
- 不依赖 DOM——可以用于 Canvas、SVG、WebGL 渲染，甚至未来的 SSR
- 支持中文、阿拉伯语、emoji 等所有语言
- API 设计精心：`prepare()` 一次预处理 + `layout()` 纯算术热路径

**怎么装**：
```bash
npm install @chenglou/pretext
```

```typescript
import { prepare, layout } from '@chenglou/pretext'
const prepared = prepare('中文文本 English mixed', '16px Inter')
const { height, lineCount } = layout(prepared, containerWidth, 20)
// 不碰 DOM，纯算术得到高度和行数！
```

**适用场景**：
- 虚拟滚动/虚拟列表（不需要 guesstimate 文本高度了）
- Masonry 布局
- Canvas/SVG 中的文本布局
- 开发期验证 UI 组件不会溢出（尤其是 AI 生成代码时）

**坑在哪**：
- 字体必须与 CSS 声明完全同步（size、weight、style、family）
- `layout()` 依赖 `prepare()` 的预处理结果，不能跳过
- 对于已经有完善虚拟化方案的成熟项目（如 react-window），迁移成本需要评估

## 七、效率/CLI 工具类

### 1. raskrebs/sonar — 一目了然看到本机跑了什么

- **GitHub**: https://github.com/raskrebs/sonar
- **Star 数**: 901 | Fork: 24 | Language: Go | License: MIT

**一句话**：一个 Go 写的 CLI，列出 localhost 上所有正在监听的端口——自动识别 Docker 容器名、Compose 项目、资源占用，还能一键 kill、tail log、shell 进容器。

**为什么不是噱头**：
- 解决了每个开发者都遇到过的问题：端口被占了，不知道是谁占的
- 比写 `lsof -iTCP -sTCP:LISTEN | grep ...` 方便一百倍
- 自动关联 Docker 容器信息——这在开发多容器项目时非常有用
- 支持 macOS 菜单栏常驻（tray app）

**怎么装**：
```bash
# macOS/Linux：
curl -sfL https://raw.githubusercontent.com/raskrebs/sonar/main/scripts/install.sh | bash

# Windows (PowerShell)：
irm https://raw.githubusercontent.com/raskrebs/sonar/main/scripts/install.ps1 | iex

# 或用 Go：
go install github.com/raskrebs/sonar@latest
```

**适用场景**：
- 本地开发环境端口管理
- Docker Compose 多容器项目的端口映射查看
- 清理残留的开发服务器进程

**坑在哪**：
- Tray app 是 Swift 写的，需要单独编译（`go install` 只装 CLI）
- 目前只有 macOS 菜单栏支持，Linux/Windows 没有 tray

### 2. hauntsaninja/git_bayesect — 用贝叶斯推理找 bug 引入的 commit

- **GitHub**: https://github.com/hauntsaninja/git_bayesect
- **Star 数**: 405 | Language: Python | License: MIT
- **HN 讨论热度**: 336 分、43 条评论

**一句话**：对 `git bisect` 的贝叶斯升级版——不需要明确的 pass/fail，只需要观察到「某个事件的发生概率变了」就能定位引入变化的 commit。

**为什么不是噱头**：
- HN 336 分——技术社区对精巧工具的认可
- 解决了 `git bisect` 的核心局限：bisect 需要每次都能确定 pass/fail，但非确定性 bug（flaky test）做不到
- 用贝叶斯推理 + Beta-Bernoulli 共轭处理未知的失败率——数学上优雅
- 可以基于文件名或 commit message 设置先验概率（prior），引导搜索方向

**怎么装**：
```bash
pip install git_bayesect
# 或：
uv tool install git_bayesect
```

```bash
git bayesect start --old $COMMIT
git bayesect fail          # 当前 commit 观察到失败
git bayesect pass --commit $COMMIT  # 指定 commit 通过
git bayesect status        # 查看分析进度
```

**适用场景**：
- 定位 flaky test 变得更 flaky 的那个 commit
- 非确定性 bug 的根因分析
- 性能退化分析（某个 commit 导致延迟概率分布改变）

**坑在哪**：
- 需要足够多的观察次数才能收敛——比 bisect 慢
- 对于确定性 bug，传统的 `git bisect` 更快更直接
- 工具还比较年轻，复杂场景（多因素变化）可能需要手动设置 prior

## 八、避坑清单

以下是近期看着很火但实际上需要谨慎的项目：

### 1. ultraworkers/claw-code — 「史上最快破 10 万 Star」

- **Star 数**: 174,005 | Fork: 105,216
- **问题**：Fork 数与 Star 数比例异常（Fork/Star > 60%），正常的开源项目这个比例通常在 5-15%。这暗示可能有刷量行为。
- **描述**：「The fastest repo in history to surpass 100K stars」——以速度为卖点的 star 记录本身就是营销信号。
- **实质**：Claude Code 的 Rust 重写版，但存在另一个仓库 `claw-code-parity` 标注为「temporary work while migration」——项目方向不稳定。
- **HN 讨论**：几乎为零。
- **结论**：刷星嫌疑极大。代码可能有参考价值，但不要因为 Star 数高就觉得它一定靠谱。

### 2. garrytan/gstack — Garry Tan 的 Claude Code 工具集

- **Star 数**: 65,529
- **问题**：作为 Y Combinator 的 CEO，Garry Tan 本人有巨大的粉丝基数。65K star 中有多少是因为「YC CEO 推荐」而非「这个工具真的好用」？
- **实质**：23 个 Markdown 文件（Claude Code skill），本质上是提示词工程。有用但不是不可替代——你可以用 10 分钟自己写类似的。
- **HN 上无人讨论这个项目本身**。
- **结论**：有用但被高估。作为学习 Claude Code skill 的范例可以参考，但不必当「必装工具」。

### 3. tanweai/pua — 「你的 AI 被放上了 PIP」

- **Star 数**: 15,264
- **问题**：名字叫「PUA」，README 写的是「你是一个曾经被寄予厚望的 P8 级工程师……你的 AI 被放上了 PIP」——这是段子，不是工具。
- **实质**：一个用心理操纵技巧来「激励」AI agent 的 Claude Code skill。
- **结论**：娱乐价值 > 实用价值。如果你在工作中用这种方式管理 AI（或人），请反思一下。

### 4. 各种 Claude Code Skill 仓库

3 月以来出现了数百个 Claude Code skill 仓库。大部分存在以下问题：
- **本质是 Markdown 文件**，不是代码——任何开发者都能写
- **缺乏维护承诺**——LLM 变了，skill 可能就废了
- **过度细分**——有人为每个小功能都开一个 repo，加剧了生态碎片化
- **质量无法保证**——没有测试、没有版本控制、没有 CI

**建议**：选 skill 时看三点——（1）是否有真实用户反馈；（2）是否持续更新；（3）是否解决你确实遇到的问题。不要因为「XXK stars」就装。

### 5. 「Rust 重写」类项目

`Kuberwastaken/claurst`、`lorryjovens-hub/claude-code-rust` 等项目声称用 Rust 重写 Claude Code，提升性能 2.5x。

- **问题**：Claude Code 的性能瓶颈在网络 I/O（API 调用），不在本地运行时。Rust 重写的实际收益存疑。
- **结论**：学 Rust 是好事，但不要以为换成 Rust 就能解决所有问题。

## 九、总结：最值得花时间学习的 3 个项目

### 第一名：karpathy/autoresearch

**为什么**：它代表了 AI 辅助研究的全新范式。不是让 AI 帮你写代码，而是让 AI 自己设计实验、执行、评估、迭代。代码极其精简（核心就三个文件），但理念的影响是深远的。即使你不跑 LLM 训练，读一遍 README 和 `program.md` 都能让你对 AI agent 的未来有新的认识。

### 第二名：googleworkspace/cli

**为什么**：它是「API 公司应该怎么做开发者工具」的标杆。运行时动态构建命令、结构化 JSON 输出、同时服务人类和 AI agent——这个设计模式会被大量复制。如果你在构建面向开发者的 API 或 CLI，这个项目的架构值得仔细研究。

### 第三名：knowsuchagency/mcp2cli

**为什么**：它解决了一个正在变得越来越严重的问题——AI agent 的 token 浪费。随着 agent 生态爆发，每次对话传一遍完整的工具 schema 是不可持续的。mcp2cli 的「把 API 调用压缩为 CLI 命令」方案是一个实用的过渡解法，而且设计简洁（零代码生成）。概念虽小但切中要害。
